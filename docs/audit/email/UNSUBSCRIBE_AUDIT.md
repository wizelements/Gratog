# UNSUBSCRIBE_AUDIT — Compliance and Suppression Flow

## Classification

**Endpoint-only classification:** Partially compliant.  
**End-to-end email-system classification:** Non-compliant.

Reason: `/api/unsubscribe` exists and can mark `newsletter_subscribers.unsubscribed=true` for tokens it understands, but current marketing/email send paths generate incompatible tokens and do not consistently read the collection that the unsubscribe endpoint writes.

## Current unsubscribe endpoint

`app/api/unsubscribe/route.ts` supports:

- `GET /api/unsubscribe?token=<signed-token>`
- `POST /api/unsubscribe { token }`
- `POST /api/unsubscribe { email }` fallback

It writes:

```text
newsletter_subscribers.updateOne(
  { email },
  { unsubscribed: true, unsubscribedAt, unsubscribedSource, updatedAt },
  { upsert: true }
)
```

Production evidence:

- `/api/unsubscribe` without token returns `400 Missing token`.
- `/unsubscribe` page returns `200`.
- `newsletter_subscribers` has 1 unsubscribed row.

## HMAC validation

### `/api/unsubscribe` token format

The route expects:

```text
base64url(email).hmac_sha256_hex(email, JWT_SECRET)
```

The MAC must be 64 hex characters.

### `lib/email/service.js` token format

The advanced email service generates:

```text
base64url(JSON { u, e, t }).hmac_sha256_hex(payload, JWT_SECRET).substring(0, 16)
```

### Current result

Tokens generated by `lib/email/service.js`, `lib/email/templates.js`, `lib/email-templates.js`, and `lib/campaign-manager.js` do **not** match the `/api/unsubscribe` verifier.

## Fallback flow

`POST /api/unsubscribe { email }` can mark an address unsubscribed without a token if the email looks syntactically valid. It returns a safe success response whether or not the email existed.

This is useful as a self-serve fallback, but there is no visible public page form for entering just an email. The `/unsubscribe` page expects a token.

## Suppression logic

| Component | Suppression source checked | Matches `/api/unsubscribe` write? |
|---|---|---:|
| `/api/unsubscribe` | writes `newsletter_subscribers.unsubscribed` | n/a |
| `lib/email/service.js canSendEmail()` | reads `unsubscribes` collection | No |
| `app/api/admin/campaigns/send/route.ts` default recipients | reads `customers.unsubscribed` | No |
| `lib/campaign-manager.js` | reads `users.emailPreferences.marketing` | No |
| `lib/resend-email.js` | no suppression check | No |

## Can unsubscribed users still receive campaigns?

**Yes.** An unsubscribe recorded in `newsletter_subscribers` is not checked by the current campaign send API or the advanced email service. If a recipient is selected through `orders.customerEmail`, a `customers` segment, or a direct send path, `newsletter_subscribers.unsubscribed=true` will not suppress it.

## Can unsubscribed users still receive transactional emails?

**Yes, intentionally.** Transactional emails such as order confirmations and admin resets are not marketing and should continue. The current problem is marketing suppression, not transactional allowance.

## Legal/compliance checks

| Requirement | Current status | Evidence |
|---|---|---|
| Working unsubscribe link in marketing email | Failing end-to-end | Current campaign/service token format differs from route verifier |
| Honor opt-out in future campaign sends | Failing | Endpoint writes `newsletter_subscribers`; senders read `customers`, `users`, or `unsubscribes` |
| Self-serve fallback | Partial | API supports POST `{ email }`, but no public UI found |
| Do not reveal subscriber existence | Pass | Safe response for valid/unknown emails |
| HMAC validation | Partial | Route verifier is strict for its own format; generators use another format |

## Exact actual unsubscribe flow today

```text
Marketing email link generated by lib/email/service.js or lib/campaign-manager.js
  ↓
/unsubscribe?token=<JSON-token-with-16-hex-signature>
  ↓
app/unsubscribe/page.js POST /api/unsubscribe { token }
  ↓
app/api/unsubscribe expects email-token-with-64-hex-signature
  ↓
Invalid token
  ↓
No unsubscribe write
```

```text
Manually valid /api/unsubscribe token or POST { email }
  ↓
newsletter_subscribers.unsubscribed=true
  ↓
Current campaign send API does not read newsletter_subscribers
  ↓
Suppression not enforced for campaigns
```

## Verdict

The unsubscribe endpoint is a good partial component, but the complete unsubscribe system is **not production-compliant** because the link generator, token verifier, write collection, and send-time suppression checks are not aligned.
